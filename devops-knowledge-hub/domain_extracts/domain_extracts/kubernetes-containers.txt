All Docker Commands

Here's a comprehensive list of commonly used Docker commands, along with their

usage:

Basic Docker Commands

1. docker version : Displays Docker version information.

2. docker info : Provides detailed information about the Docker installation.

3. docker --help : Lists all available Docker commands and options.

Image Management Commands

1. docker pull <image> : Downloads an image from a Docker registry (e.g., Docker

Hub).

○ Example: docker pull nginx

2. docker images : Lists all Docker images available on the system.

○ Example: docker images

3. docker rmi <image> : Deletes a Docker image from the system.

○ Example: docker rmi nginx

4. docker build -t <name> <path> : Builds a Docker image from a Dockerfile.

○ Example: docker build -t myapp:latest .

5. docker tag <source_image> <target_image> : Tags an image with a new

name.

○ Example: docker tag nginx:latest myrepo/nginx:v1

6. docker save -o <file> <image> : Saves an image to a tar archive.

○ Example: docker save -o nginx.tar nginx:latest

7. docker load -i <file> : Loads an image from a tar archive.

○ Example: docker load -i nginx.tar

Container Management Commands

1. docker run <image> : Creates and starts a new container from an image.

○ Example: docker run nginx

2. docker run -d <image> : Runs a container in detached mode (in the

background).

○ Example: docker run -d nginx

3. docker run -it <image> : Runs a container interactively with a terminal.

○ Example: docker run -it ubuntu bash

4. docker ps : Lists all running containers.

○ Example: docker ps

5. docker ps -a : Lists all containers, including stopped ones.

○ Example: docker ps -a

6. docker stop <container> : Stops a running container.

○ Example: docker stop my_container

7. docker start <container> : Starts a stopped container.

○ Example: docker start my_container

8. docker restart <container> : Restarts a container.

○ Example: docker restart my_container

9. docker rm <container> : Deletes a stopped container.

○ Example: docker rm my_container

10. docker exec -it <container> <command> : Executes a command in a

running container.

○ Example: docker exec -it my_container bash

11. docker logs <container> : Displays logs from a container.

○ Example: docker logs my_container

12. docker attach <container> : Attaches to a running container's console.

○ Example: docker attach my_container

13. docker kill <container> : Forcefully stops a container.

○ Example: docker kill my_container

Container Networking Commands

1. docker network ls : Lists all Docker networks.

○ Example: docker network ls

2. docker network create <name> : Creates a new Docker network.

○ Example: docker network create my_network

3. docker network rm <name> : Deletes a Docker network.

○ Example: docker network rm my_network

4. docker network connect <network> <container> : Connects a container to

a network.

○ Example: docker network connect my_network my_container

Know your Kubernetes Components
Cluster: 
A Kubernetes cluster is a collection of nodes (machines) where
Kubernetes manages and orchestrates containerized applications. A
cluster represents a Kubernetes deployment as a whole, combining
both worker nodes (where applications run) and a control plane (which
manages and monitors the cluster).
Control Plane: 
The control plane is responsible for the cluster’s overall management.
The cluster’s control plane continuously monitors the actual state of
the applications and resources, adjusting to match the desired state.
This "desired state" is defined by the configurations (usually in YAML
files) you apply to Kubernetes.
Control Plane comprises of several components that work together to
manage cluster operations:
➲ API Server: Acts as the main access point for Kubernetes, handling
communication between the user and the control plane.
➲ Controller Manager: Manages controllers, which ensure that the
actual state matches the desired state (like ensuring the right number
of pods).
➲ Scheduler: Assigns new pods to nodes based on available resources
and other factors.
➲ etcd: A distributed key-value store that holds configuration data,
cluster state, and metadata. It is similar to a database. It has the actual
status of the pod.

Complete Guide to Dockerfile Parameters 🐳 🚀
A Dockerfile is a script containing a set of instructions to build a Docker image. It automates
the creation of lightweight, portable, and consistent containerized applications. Below is a
detailed breakdown of all important Dockerfile parameters, their syntax, and examples.
  FROM (Base Image)
Defines the base image for the container.
📌 Every Dockerfile must start with FROM.
✅ Syntax: FROM <image>:<tag>
✅ Example: FROM ubuntu:20.04
(Uses Ubuntu 20.04 as the base image.)
  LABEL (Metadata)
Adds metadata (author, description, version, etc.) to an image.
✅ Syntax: LABEL key="value"
✅ Example: LABEL maintainer="John Doe"
LABEL description="This is a sample web application"
  WORKDIR (Set Working Directory)
Sets the working directory for subsequent commands.
✅ Syntax: WORKDIR /app
✅ Example: WORKDIR /usr/src/app
(All commands like RUN, CMD, COPY, etc., will now run inside /usr/src/app.)
  COPY (Copy Files from Host to Container)
Copies files from the local system to the container image.
✅ Syntax: COPY <source> <destination>
✅ Example: COPY index.html /usr/share/nginx/html/
(Copies index.html from the host to the Nginx container’s web directory.)
  ADD (Similar to COPY, but More Powerful)
Like COPY, but also extracts compressed files (.tar, .zip, etc.) and supports remote URLs.
✅ Syntax: ADD <source> <destination>
✅ Example: ADD app.tar.gz /usr/src/app/
(Extracts app.tar.gz into /usr/src/app/ inside the container.)
  RUN (Execute Commands in the Image Build Process)
Runs a shell command while building the image.
✅ Syntax: RUN <command>
✅ Example: RUN apt-get update && apt-get install -y nginx
(Updates package lists and installs Nginx in the image.)
  CMD (Define Default Container Startup Command)
Specifies the default command to run inside the container at runtime.
✅ Syntax: CMD ["executable", "param1", "param2"]
✅ Example: CMD ["nginx", "-g", "daemon off;"]
(Starts Nginx when the container runs.)
🛑 Difference between CMD and RUN
RUN executes during build time (inside the image).
CMD executes when the container starts (at runtime).
  ENTRYPOINT (Define a Fixed Executable for the Container)
Similar to CMD, but it cannot be overridden when running a container.
✅ Syntax: ENTRYPOINT ["executable", "param1", "param2"]
✅ Example: ENTRYPOINT ["/bin/bash", "-c"]
(Forces the container to run Bash shell when started.)
🛑 Difference between CMD and ENTRYPOINT
CMD can be overridden when running the container.
ENTRYPOINT always executes, even if arguments are provided.
  EXPOSE (Define Container Listening Port)
Declares which port the application will use inside the container.
✅ Syntax: EXPOSE <port>
✅ Example: EXPOSE 80
(Tells Docker that the application inside uses port 80.)
🚀 To publish a port while running the container: docker run -p 8080:80 my-image
(Maps port 80 in the container to port 8080 on the host.)

Docker 
Docker is a platform that allows developers to package applications into containers. 
Containers are lightweight, portable, and include everything an application needs to 
run—like code, libraries, and system tools—so it can work consistently across different 
environments. 
Think of Docker as a shipping container for software. Just like a shipping container 
holds goods and can be transported anywhere, Docker containers hold software and can 
run consistently on any system that supports Docker. 
Docker Image 
A Docker image is like a blueprint or recipe for a container. It includes: 
• The application code. 
• Necessary files and dependencies. 
• Instructions on how to set up and run the application. 
You can think of it as a frozen snapshot of the environment needed to run your app. 
Images are static, meaning they don’t change. 
Docker Container 
A Docker container is a running instance of a Docker image. If the image is the 
blueprint, the container is the actual building created from that blueprint. Containers are: 
• Isolated: They don’t interfere with other containers or the host system. 
• Lightweight: They share the host machine’s kernel, so they’re much smaller than 
virtual machines. 
• Ephemeral: They can be stopped, restarted, or removed easily without affecting 
the Docker image. 
DevOps and Docker 
In DevOps, Docker is widely used to: 
• Streamline development: Developers can ensure the app works the same way in 
development, testing, and production. 
• Speed up deployment: Docker containers are lightweight, so they start quickly 
and are easy to manage. 
• Enable CI/CD pipelines: Docker integrates seamlessly with tools like Jenkins, 
GitLab, and Azure DevOps to automate testing, building, and deployment.

Simplified Analogy for Docker 
• Docker image: A recipe book for your favorite dish. 
• Docker container: The actual dish prepared using that recipe. 
• Docker: The kitchen setup that lets you cook the dish consistently anywhere. 
Key Benefits for DevOps 
1. Consistency: Avoids "works on my machine" problems. 
2. Portability: Runs anywhere Docker is supported. 
3. Scalability: Easily run multiple containers for large applications. 
4. Resource efficiency: Uses fewer system resources than traditional virtual 
machines. 
1. Consistency: Avoids "Works on My Machine" Problems 
• Problem: In traditional software development, an application may work perfectly 
on a developer's local environment but fail when moved to a testing or production 
environment due to differences in configurations, dependencies, or operating 
systems. 
• Solution with Docker: Docker containers encapsulate the application along with 
its dependencies (libraries, system tools, etc.), ensuring the application runs 
identically in development, testing, and production. 
• Example: If a Python app uses a specific version of a library, Docker ensures that 
exact version is used everywhere the app runs. This eliminates surprises caused by 
different environmen

Q1: What is Docker, and how does it differ from virtual machines (VMs)? 
• Purpose : Tests foundational understanding of containerization vs. virtualization. 
• Expected Answer : 
• Docker uses containers to package applications with their dependencies, running them in 
isolated environments on the host OS. 
• Unlike VMs, which include a full OS and hypervisor, Docker shares the host OS kernel, 
making it lightweight and faster to start/stop. 
• Key differences: Resource usage, startup time, isolation level, and portability.

Q2: Explain the difference between an image and a container in Docker. 
• Purpose : Assesses basic Docker terminology and concepts. 
• Expected Answer : 
• An image is a read-only template containing the application code, libraries, and dependencies. 
• A container is a runtime instance of an image—a writable layer where the application runs. 
• Example: docker run creates a container from an image.

• Expected Answer : 
• A Dockerfile is a script containing instructions to build a Docker image. 
• Key components: 
• FROM: Base image. 
• RUN: Execute commands during image build. 
• CMD/ENTRYPOINT: Define default commands for the container. 
• COPY/ADD: Copy files into the image. 
• EXPOSE: Specify ports to expose. 
• ENV: Set environment variables.

Q4: What is a multi-stage build in Docker, and why is it useful? 
• Purpose : Tests optimization techniques for Docker images. 
• Expected Answer :

• Benefits:

=== SOURCE: Docker file Components Dockerfile Instructions: FROM: Defines the base image… ===
Docker file Components 
Dockerfile Instructions:

FROM: Defines the base image for your Docker image. This is 
essential as it sets the foundation for your environment. 
COPY: Copies files and directories from your Docker host (your 
computer) into the container's filesystem. 
ADD: Similar to COPY, but with added functionality:

Downloads files from URLs and adds them to the container.

Automatically extracts compressed files (e.g., .tar, .zip) when added. 
RUN: Executes commands within the container during the image 
building process. Used for installing packages, setting up 
configurations, and other build-time tasks. 
Container Execution: 
CMD: Specifies the default command to run when a container is 
created from the image. Think of it as the "main" process of your 
container. 
ENTRYPOINT: Similar to CMD, but with higher priority. It's more 
difficult to override, making it suitable for defining the primary 
executable of your container. ENTRYPOINT will take precedence over 
CMD. 
WORKDIR: Sets the working directory inside the container. 
Subsequent commands will be executed from this directory. 
Metadata &amp; Configuration: 
LABEL: Adds metadata to your Docker image. Useful for providing 
descriptions, version information, or any relevant details. 
EXPOSE: Documents which ports the container will listen on at 
runtime. It does not publish the ports but serves as documentation 
for port mapping. 
AUTHOR: Specifies the author of the Dockerfile. 
ARG: Defines build-time variables. These variables are only 
accessible during the image building process and are not available 
within the container. 
ENV: Sets environment variables that persist within the container. 
These variables can be accessed by applications running inside the 
container.

Important Note: `LABEL`, `AUTHOR`, and `EXPOSE` are primarily 
used for documentation purposes. They provide information about 
the image but do not directly affect its functionality.

First Understand the Problem Statement (Understanding the Need for 
Ingress) 
Imagine you have a Kubernetes cluster running multiple services, such as a frontend that handles 
user interactions, a backend that processes requests, and a database that stores information. To 
make these services accessible to users outside the cluster, you need to expose them. 
One way to expose services is by using NodePort, which assigns a unique port on every node. 
However, this approach has a major drawback —each service requires an open port, increasing 
security risks by making cluster nodes directly accessible from the internet. Another option is using 
a LoadBalancer, which provides external access by creating a dedicated cloud-based load balancer 
for each service. While this method is effective, it becomes expensive and difficult to manage as 
the number of services grows. 
Managing multiple NodePorts or LoadBalancers for different services leads to security 
vulnerabilities, increased operational complexity, and high costs. Instead of exposing every service 
separately, a more efficient approach is needed. 
And the solution? Ingress! 
Ingress acts as a single entry point that routes external requests to the appropriate services based 
on defined rules. Instead of assigning a separate LoadBalancer or NodePort for each service, 
Ingress allows controlled and secure access while simplifying traffic management, reducing costs, 
and improving security.

Prerequisites Before Using Ingress in Your Cluster 
Before implementing Ingress to manage external traffic in your Kubernetes cluster, there are two 
essential components you must have in place. Without these, Ingress cannot function properly.

• Ingress Controller – This is the actual component that processes incoming traffic and 
enforces the routing rules defined in Ingress Resources. Kubernetes does not come with an 
Ingress Controller by default, so you need to install and configure one separately. 
• Ingress Resource – This is a Kubernetes object that defines the rules for routing traffic, such 
as which requests should be sent to which service based on domain names or URL paths. 
However, an Ingress Resource alone does not handle traffic; it requires an Ingress Controller 
to function. 
Without an Ingress Controller, Ingress Resources are just configurations that Kubernetes cannot 
act upon. In the next section, we will dive deeper into both Ingress Controller and Ingress Resource 
to understand how they work together to manage external traffic efficiently.

Hand-Written Notes

On Kubernetes Resource Quota

(A Simple strong Student Guide)

What is Resource Quota? 
Resource Quota is a method in Kubernetes to control resource usage at the project level 
(Namespace level). 
There are two types of Resource Quota: 
1. Computing Level Quota – Controls CPU and RAM (Memory) usage. 
2. Object Level Quota – Limits the number of Pods, Deployments, Services, PVCs, etc.

Why is Resource Quota Applied at the Namespace Level? 
In Kubernetes, every resource is created inside a Namespace. A Namespace defines the boundary 
for project resources. So, Resource Quota is always set at the Namespace level to manage resource 
limits effectively.

Why is Resource Quota Important? 
Imagine a Kubernetes Cluster where multiple teams work together. If one team uses too many 
resources, other teams may not get enough . So Resource Quota prevents this by ensuring fair 
distribution. 
Problem: One team uses all the resources. 
Solution: Quota limits resource usage per team. 
Example: 
• Team A (Amazon) gets limited resources. 
• Team B (Facebook) gets its own fair share.

Namespaces and Resource Quota 
In Kubernetes, a Namespace divides a cluster into smaller sections, providing logical isolation at 
the project level. When a Resource Quota is applied to a Namespace, it ensures that all resources 
within that Namespace adhere to the defined limits. However, it does not impact other projects, as 
each operates in its own separate Namespace. 
For example, if a cluster has a total capacity of 32GB RAM and 16 CPU Cores, the administrator can 
allocate resources as follows: 
• Team A gets 20GB RAM and 10 CPU Cores. 
• Team B gets 10GB RAM and 4 CPU Cores. 
• 2GB RAM and 2 CPU Cores are reserved for future use.

Example: 
Team A → Namespace = amazon 
Team B → Namespace = facebook 
Each team can create resources only within their assigned Resource Quota, ensuring fair 
distribution of cluster resources.

Types of Resource Quota 
As discussed earlier, there are two main types of Resource Quota: 
1. Computing Quota – Controls CPU and Memory (RAM). 
2. Object Quota – Limits the number of Kubernetes objects like Pods, Deployments, Services, 
PVCs, etc.

Type-1 (Object Resource Quota) – Kubernetes object Resources 
How Does Object Quota Work? 
A Object Quota sets a limit on the maximum resources that can be used in a Namespace. It 
controls computing resource like CPU, RAM, storage, and the number of objects (Pods, Services, 
Deployments, etc.). 
Example: Setting a Object Quota for the "AMAZON" Namespace 
apiVersion: v1 
kind: ResourceQuota 
metadata:

name: team-a-quota 
spec:

hard:

pods: "10"

# Maximum 10 Pods

services: "5" # Maximum 5 Services

deployments: “10”

# Maximum 10 deployments

secrets: "15" # Maximum 15 Secrets

requests.cpu: "2"

# Max 2 CPU requests

requests.memory: "4Gi"

# Max 4GB RAM requests

limits.cpu: "4"

# Max 4 CPU usage

limits.memory: "8Gi"

# Max 8GB RAM usage

requests.storage: "100Gi"

# Max 100GB Storage

Easy Notes on Service, kube-proxy & CNI

What is a service ? 
A Service in Kubernetes helps expose an application running in your cluster by providing a single, 
stable network endpoint. This ensures that even if your application runs in multiple Pods (which may 
change dynamically), clients can always access it reliably.

Why Do We Need a Service? 
1. Pods Are Temporary (Ephemeral) – Kubernetes creates and destroys Pods as needed, so 
their IP addresses change dynamically. 
2. Consistent Access – If an application has multiple backend Pods, a Service provides a fixed 
way to reach them. 
3. Load Balancing – Services distribute traffic among multiple Pods automatically.

How a Service Works? 
• A Service groups a set of Pods using labels and selectors. 
• It gives them a stable network identity (ClusterIP) instead of IP.

• Clients can use this identity to interact with the backend Pods. 
For example, assume we have a backend application running in 3 Pods. Instead of connecting to 
individual Pods, we create a Service that directs traffic to all healthy backend Pods.

Types of Services 
1. ClusterIP (Default) 
• Exposes the Service only within the cluster. 
• Not accessible from outside. 
• Example: Internal communication between microservices. 
Example YAML: 
apiVersion: v1 
kind: Service 
metadata:

name: my-service 
spec:

selector:

app: my-backend

ports:

- protocol: TCP

port: 80

targetPort: 9376

Here, all Pods with label app=my-backend will be reachable at port 80 via the Service.

2. NodePort 
• Exposes the Service on a specific port of each Node. 
• Accessible from outside the cluster using NodeIP:NodePort. 
• Useful for debugging or basic external access. 
Example YAML: 
apiVersion: v1 
kind: Service 
metadata:

name: my-service 
spec:

type: NodePort

selector:

app: my-backend

ports:

- port: 80

targetPort: 9376

nodePort: 30007 # This port is accessible on every node

If the cluster node has IP 192.168.1.10, then you can access the service at: 
http://192.168.1.10:30007

3. LoadBalancer 
• Used when running Kubernetes on cloud providers (AWS, GCP , Azure). 
• It provisions an external load balancer to distribute traffic. 
• Suitable for public-facing applications. 
Example YAML: 
apiVersion: v1 
kind: Service 
metadata:

name: my-service 
spec:

type: LoadBalancer

selector:

app: my-backend

ports:

- port: 80

targetPort: 9376

• If the cloud provider assigns 192.0.2.127 as the external IP , users can access it via: 
http://192.0.2.127

4. ExternalName 
• Maps a Service to an external DNS name. 
• No traffic is routed inside the cluster; instead, it resolves to an external resource. 
Example YAML: 
apiVersion: v1 
kind: Service 
metadata:

name: my-database 
spec:

type: ExternalName

externalName: db.example.com

When applications inside the cluster query my-database, they get redirected to 
db.example.com.

How to Kubernetes talk with pods

Kubernetes facilitates communication with and between pods using a combination of networking 
principles, resources, and configurations. Here's a detailed explanation of how Kubernetes enables 
pods to communicate: 
1. Pod-to-Pod Communication 
• Scenario: Applications in different pods need to interact, e.g., a frontend pod communicates 
with a backend pod. 
• How It Works: 
o Each pod gets a unique IP address within the cluster. 
o Kubernetes uses an overlay network (provided by CNI plugins like Calico, Flannel, or 
Weave Net) to ensure all pods can communicate directly with each other using their IP 
addresses. 
o Pods can reach other pods using their ClusterIP if they're in the same namespace or 
across namespaces with the proper configuration. 
• Challenges Solved: 
o Direct communication across nodes in a distributed system. 
o Uniform IP addresses for pods, even if they're recreated. 
2. Pod Communication via Services 
• Scenario: A frontend pod communicates with a backend pod using a stable endpoint. 
• How It Works: 
o Kubernetes services provide a stable DNS name and IP address for pods, even as pods 
are dynamically created or destroyed. 
o Types of Services: 
▪ ClusterIP (default): Provides an internal cluster IP for communication within 
the cluster. 
▪ NodePort: Exposes the service on a static port of each node in the cluster. 
▪ LoadBalancer: Uses a cloud provider's load balancer to expose the service 
externally. 
▪ ExternalName: Maps the service to an external name (DNS).

This routes traffic to pods labeled app: backend on port 
8080.

3. Inter-Pod Communication Across Namespaces 
• Scenario: Pods in different namespaces need to communicate. 
• How It Works: 
o Use the Service's fully qualified domain name (FQDN), such as <service-
name>.<namespace>.svc.cluster.local. 
o Example: A frontend in the default namespace communicates with a backend 
service in the production namespace using 
backend.production.svc.cluster.local.

• Scenario: An external client or user needs to access a pod. 
• How It Works: 
o Use Ingress: Manages external HTTP and HTTPS traffic routing to services in the 
cluster.

world.

Basic Kubernetes Questions 
1. What is Kubernetes? 
Kubernetes is an open-source container orchestration system for automating application deployment, 
scaling, and management. 
2. What are the main components of Kubernetes? 
o Master Node: API Server, Controller Manager, Scheduler, etcd 
o Worker Node: Kubelet, Kube Proxy, Container Runtime 
3. What is a Pod in Kubernetes? 
A Pod is the smallest deployable unit in Kubernetes, encapsulating one or more containers with 
shared storage and networking. 
4. What is the difference between a Deployment and a StatefulSet? 
o Deployment: Manages stateless applications. 
o StatefulSet: Manages stateful applications with stable network identities and persistent 
storage. 
5. What is a Service in Kubernetes? 
A Service provides a stable network endpoint to expose Pods, enabling load balancing and inter-Pod 
communication. 
Kubernetes Architecture & Components 
6. What is etcd in Kubernetes? 
etcd is a distributed key-value store used to maintain cluster state and configuration.

7. What is the role of the API Server? 
The API Server acts as the frontend for Kubernetes, handling REST requests and communicating 
with etcd. 
8. What is the Kubelet? 
The Kubelet is an agent running on worker nodes, ensuring that containers are running as defined in 
Pod specifications.

Intro to Kubernetes 
In order to set the stage to dive deep into Kubernetes terms it helps if we 
understand some terms that aren't Kubernetes specific, but lay the groundwork 
for deploying a Kubernetes cluster.

cgroup (control group) 
Cgroups are a Linux kernel feature. Combined with another Linux kernel 
feature, namespaces, you can isolate processes from each other. That is 
basically how containers are created. Cgroups specifically limit, account for, 
and isolates the resource usage of a collection of processes.

Cloud Native Computing Foundation (CNCF) 
The Cloud Native Computing Foundation is part of the nonprofit Linux 
Foundation. According to their site they support the growth and health of 
cloud native open source software. A few of the projects under CNCF include 
Kubernetes, Prometheus, and Envoy. Google is one of the top contributors to 
CNCF.

Container 
Containers are often compared to virtual machines (VMs) as they both offer a 
way to package an application together with libraries and other d ependencies. 
However, instead of virtualizing the hardware stack like VMs, containers do it 
at the operating system level. Containers wrap the code of an application as 
well as the other dependencies it needs when running. An app container is 
specifically used for running an application as opposed to an init container that 
runs separately with initialization instructions for an entire workload.

Container Runtime 
The container runtime is software that runs the containers. Kubernetes 
supports several contai ner runtimes: Docker, containerd, CRI -O, and any 
implementation of the Kubernetes CRI (Container Runtime Interface). The 
most popular container runtime today is Docker.

Containerd 
An open-source container runtime that is part of CNCF. It is a daemon for 
Linux and Windows that manages the container lifecycle of its host system. It 
takes care of image transfer and storage, container execution and supervision, 
providing network access, and more. You can read more about it 
here https://containerd.io/ .

Docker 
Is the most popular container runtime, a software that runs containerization of 
applications. Using the resource isolation features of the Linux kernel, 
namespaces and cgroups, Docker allows independent containers to run within 
a single Linux instance. When Kubernetes schedules a pod to a node, Docker 
is instructed by the kubelet running on that node to launch the containers.

Microservices 
This is a strategy for developi ng software where an application is organized as 
a group of loosely connected services. It is focused on building simpler 
modules that make up the entirety of a software application. Implementing 
microservices allows changes that need to be made to be done in smaller 
chunks where some of the modules don’t even need to be touched. You can 
also scale specific pieces of an application instead of the entire thing.

Introduction to Docker Containers
What is Docker?
Docker is an open-source platform designed to simplify the process of
developing, shipping, and running applications. It uses containerization
technology to package applications and their dependencies into lightweight,
portable containers. These containers are isolated from each other and the
host system, ensuring consistency across different environments.
Why Use Docker?
Isolation: Each container runs in its own isolated environment,
preventing conflicts between applications.
Portability: Containers can run on any system that supports Docker,
making it easy to move applications between development, testing, and
production environments.
Efficiency: Containers share the host system’s kernel, making them
more lightweight and faster to start compared to virtual machines.
Scalability: Docker makes it easy to scale applications horizontally by
running multiple container instances.
Key Components of Docker
Docker Engine: The core component that builds and runs containers.
Docker Images: Read-only templates used to create containers.
Docker Containers: Runnable instances of Docker images.
Docker Hub: A cloud-based repository for sharing Docker images.

What is Ingress?
Ingress is a Kubernetes API object used to manage and control access to services within a cluster. It acts as an intelligent router
for HTTP and HTTPS traffic, allowing you to expose multiple services through a single IP address or LoadBalancer. Instead of
exposing every service with a dedicated LoadBalancer or NodePort, you can configure an Ingress to direct traffic based on rules
such as hostnames or URL paths.
This is especially useful in microservices architecture, where applications are broken down into multiple services that need to be
accessed externally.

Ingress consists of two critical parts that work together to handle traffic routing effectively:
1️⃣ Ingress Resources (Routing Rules)
These are Kubernetes API objects where you define the desired routing behavior for your cluster:
A Frontend Pod (UI) that users interact with•
A Backend Pod that handles APIs and business logic•
A Database Pod used for internal storage and not meant to be accessed by users•

Route example.com/backend to the backend service•

Ingress Resources are declarative and can be managed using YAML manifests. They tell the Ingress Controller how to route traffic,

2️⃣ Ingress Controllers
The Ingress Controller is the actual component that reads the rules defined in Ingress Resources and implements the routing. It
watches for changes in the Ingress Resources and dynamically configures the load balancing behavior.

Kubernetes Services

In Kubernetes, Services are an abstraction layer that defines a logical set of Pods and provides a 
stable endpoint (IP and DNS) to access them. Services enable communication between different 
components of an application and allow external access to your application. 
Here's an elaboration on Kubernetes Services: 
Why Use Services? 
Pods in Kubernetes are ephemeral; they can be created, destroyed, and replaced. Each Pod gets its 
own IP address, which changes when the Pod is recreated. Services provide a way to reliably access 
Pods even as their IP addresses change. 
Types of Services 
1. ClusterIP (Default): 
o Exposes the Service within the Kubernetes cluster. 
o The Service is accessible only from other Pods in the cluster. 
o Example use case: Backend services accessed only by other application components. 
EXAMPLE :-

NodePort: 
• Exposes the Service on a static port (30000-32767) on each node in the cluster. 
• Allows external traffic to access the Service through <NodeIP>:<NodePort>. 
• Example use case: Basic external access during development.

LoadBalancer: 
• Automatically provisions an external load balancer (e.g., AWS ELB, GCP Load Balancer) to 
expose the Service. 
• Routes external traffic to the Service. 
• Example use case: Exposing production-grade applications to the internet.

ExternalName: 
• Maps a Service to an external DNS name. 
• Does not define selectors or manage Pods but allows Kubernetes applications to use an 
external service seamlessly. 
• Example use case: Connecting to an external database.

How Services Work Internally 
1. Kube-Proxy: 
o Each node runs a component called kube-proxy that implements virtual IPs for 
Services and routes traffic to the appropriate backend Pods. 
2. Endpoints Object: 
o Kubernetes creates an Endpoints object for each Service, listing the Pod IPs that 
match the Service's selector. 
3. Service Discovery: 
o Services are accessible via their DNS names (e.g., my-
service.default.svc.cluster.local) within the cluster. 
Advanced Concepts 
1. Headless Services: 
o Set clusterIP: None to skip load balancing and DNS resolution. Useful for direct 
communication with Pods (e.g., in StatefulSets).

Kubernetes Services 
Kubernetes Services are a way to expose and manage communication between different parts of your 
application and make it accessible to users or other applications. They ensure that your application 
components can reliably talk to each other, even when individual Pods (smallest units in Kubernetes) 
are constantly being created or destroyed. 
Key Points About Kubernetes Services (in simple words): 
What is a Service? 
A Service is like a bridge that connects your app to the outside world or other parts of the app within 
the Kubernetes cluster. 
Why is it needed? 
Pods in Kubernetes can come and go (due to scaling or updates), and their IP addresses keep 
changing. A Service gives them a fixed way to communicate. 
How does it work? 
It uses labels to find the right Pods and directs traffic to them, even if their IP addresses change. 
Types of Services: 
ClusterIP (default): Makes the app accessible only inside the cluster. Good for internal 
communication. 
NodePort: Opens a specific port on every node, making the app accessible outside the cluster. 
LoadBalancer: Uses a cloud provider's load balancer to make the app accessible externally with a 
public IP. 
ExternalName: Maps the Service to an external DNS name. 
Real-life example: 
Think of Pods as chefs in a restaurant (kitchen). A Service is like a waiter who takes orders (requests) 
from customers and delivers them to the right chef, even if the chef changes during the day.

1. ClusterIP (default) Purpose:

running inside the cluster.

How it works:

Use Case:

How it works:

Use Case:

1. Scenario: You want to deploy an application with multiple replicas.
How do you scale your Kubernetes deployment?
Answer: You can scale a Kubernetes deployment by using the kubectl scale command or
by modifying the deployment's YAML file.
Using kubectl command:
kubectl scale deployment <deployment-name> --replicas=<number-of-replicas>
For example:
kubectl scale deployment my-app --replicas=5
Alternatively, you can edit the YAML file of the deployment and change the replicas
field:
apiVersion: apps/v1
kind: Deployment
metadata:

name: my-app
spec:

replicas: 5
Apply the changes:
kubectl apply -f deployment.yaml
2. Scenario: You have a Kubernetes pod that is stuck in
CrashLoopBackOff. How do you diagnose and fix it?
Answer:
1. Check pod logs to understand why it’s crashing:
kubectl logs <pod-name> --previous
2. Describe the pod to get detailed information about its state and events:
kubectl describe pod <pod-name>
3. Look for any error messages or failed container states in the logs. Common
issues include misconfigurations, missing files, or incorrect image versions.
4. Fix the issue based on the logs (e.g., fix environment variables, update the
image, or modify the configuration) and then restart the pod.
To restart the pod manually:
kubectl delete pod <pod-name>

Multi-stage Docker Build

Multi-stage Docker Build is a technique in Docker that enables you to create smaller 
and more efficient container images by dividing the build process into multiple stages. 
Each stage is defined within the same Dockerfile, but only the final stage contributes 
to the resulting image. This approach is particularly useful for eliminating unnecessary 
build dependencies and intermediate files from the final image.

Key Benefits: 
1. Smaller Images: Keeps the final image lightweight by excluding unnecessary 
dependencies and tools used during the build process. 
2. Security: Reduces the attack surface by not including sensitive or unnecessary 
files in the final image. 
3. Simplicity: Combines multiple build steps in a single Dockerfile, improving 
maintainability.

Explanation with Example: 
Let's assume you have a simple application written in Go. 
Application Structure: 
• main.go: The source code. 
• Dockerfile: The Docker build configuration.

Step-by-Step Breakdown: 
1. Stage 1: Build Stage (builder) 
o A base image with all build tools (e.g., golang:1.20) is used. 
o Source code (main.go) is copied into the container. 
o The application is compiled (go build), creating the main binary. 
2. Stage 2: Final Image 
o A minimal base image (alpine) is used for the final container. 
o The compiled binary (main) is copied from the builder stage. 
o The final image contains only the application binary and necessary files, 
excluding the Go compiler and build tools.

Build and Run the Docker Image: 
1. Build the image

2. Run the container

Why Use Multi-Stage Builds? 
• Efficiency: Intermediate layers are discarded, keeping the final image lean. 
• Clean Separation: Separate build and runtime environments. 
• Versatility: Combine multiple stages for complex workflows (e.g., building assets 
in one stage and the app in another).

Understanding Ingress in Kubernetes: Key Concepts and Configuration

Introduction: 
Ingress in Kubernetes is a vital resource for managing external access to services within a cluster. It 
provides a streamlined way to route HTTP and HTTPS traffic to various services based on criteria 
such as URL paths and hostnames. By using Ingress, you can consolidate multiple services under a 
single external IP address, simplifying traffic management and reducing the need for multiple Load 
Balancers or NodePorts. This guide will walk you through the essential concepts of Ingress, its 
benefits, and how to set up an NGINX Ingress Controller to handle traffic routing effectively. You will 
learn about different types of Ingress rules, such as path-based and host-based routing, and how to 
configure SSL termination for secure connections. 
This is a demonstration of how to use Kubernetes Ingress to route traffic to different services in your 
cluster based on different paths. 
What is Ingress? 
Ingress in Kubernetes is a resource that manages external access to services within a cluster, typically 
HTTP or HTTPS traffic. It allows you to define rules for routing traffic to different services based on 
URL paths, hostnames, or other criteria, without exposing multiple services to the outside world 
directly. Essentially, Ingress acts as a gateway that directs incoming requests to the appropriate 
services within the cluster. 
Why Ingress? 
Imagine you run multiple web applications (e.g., a blog, an e-commerce site, and an API) on the 
same Kubernetes cluster. Without Ingress, you’d need to expose each service with its own 
LoadBalancer or NodePort, which can be expensive and complex to manage. Ingress simplifies this 
by allowing you to route traffic to the correct service based on the URL path or hostname, all under a 
single external IP address.

Types of Ingress Rules: 
1. Path-Based Routing: Routes traffic to different services based on the URL path. 
• Example: If you run an e-commerce site, traffic to yourdomain.com/courses might go to the shopping service, while 
traffic to yourdomain.com/blogs goes to the blog service.

2. Host-Based Routing: Routes traffic based on the hostname. 
• Example: Traffic to https://exams.mycourses.com goes to your shopping service, while traffic 
to https://books.mycourses.com goes to your blog service.

3. SSL Termination: Handles HTTPS traffic by terminating SSL/TLS at the Ingress, meaning the Ingress manages the encryption 
and forwards the traffic to services as plain HTTP . 
• Example: You can secure your services with HTTPS by managing certificates at the Ingress level, offloading the 
complexity of SSL management from the individual services.

Kubernetes is an open source orchestration tool developed by Google for managing 
microservices or containerized applications across a distributed cluster of nodes.

Kubernetes provides highly resilient infrastructure with zero downtime deployment capabilities, 
automatic rollback, scaling, and self -healing of containers (which consists of auto -placement,

Kubernetes created from Borg & Omega projects by google as they use it to orchestrate they data

Google open -sourced kubernetes at 2014.

• Allocation of HW resources between containers.

• Load balancing, traffic routing and service discovery of containers.

• Health monitoring of containers.

• Securing the interactions between containers.

The Three Pillars of Kubernetes Troubleshooting
There are three aspects to effective troubleshooting in a Kubernetes cluster: 
understanding the problem, managing and remediating the problem, and 
preventing the problem from recurring.
Understanding
In a Kubernetes environment, it can be very difficult to understand what 
happened and determine the root cause of the problem. This typically involves:
Reviewing recent changes to the affected cluster, pod, or node, to see what 
caused the failure.
Analyzing YAML configurations, Github repositories, and logs for VMs or bare 
metal machines running the malfunctioning components.
Looking at Kubernetes events and metrics such as disk pressure, memory 
pressure, and utilization. In a mature environment, you should have access to 
dashboards that show important metrics for clusters, nodes, pods, and 
containers over time.
Comparing similar components behaving the same way, and analyzing 
dependencies between components, to see if they are related to the failure.
To achieve the above, teams typically use the following technologies:
Monitoring Tools: Datadog, Dynatrace, Grafana, New Relic
Observability Tools: Lightstep, Honeycomb
Live Debugging Tools: OzCode, Rookout
Logging Tools: S plunk, LogDNA, Logz.io

what is different between docker and ECR

Docker and Amazon ECR (Elastic Container Registry) serve different purposes in the world of 
containerization. Here's the key difference between them:

1. Docker: 
• Purpose: Docker is a platform for developing, shipping, and running applications inside containers. A 
container is a lightweight, portable, and self-sufficient unit that can package up an application and all 
its dependencies. 
• Key Functions: 
o Build and Package Containers: Docker allows you to create (build) container images from 
Dockerfiles and package your applications into containers. 
o Run Containers: Docker runs containers on your local machine or server, helping developers 
easily test and deploy their applications in a consistent environment. 
o Docker Hub: Docker Hub is a public registry for Docker images where you can upload, share, 
and download pre-built container images. 
2. Amazon ECR (Elastic Container Registry): 
• Purpose: Amazon ECR is a fully managed container image registry provided by AWS (Amazon Web 
Services). It is a service for storing and managing Docker container images in a secure, scalable way 
within the AWS ecosystem. 
• Key Functions: 
o Image Storage: ECR stores Docker container images (built using Docker or other tools), and 
provides a secure and scalable repository for those images. 
o Private Repositories: It allows you to create private repositories where you can store your 
custom images. 
o Integration with AWS Services: ECR integrates seamlessly with AWS services like Amazon 
ECS (Elastic Container Service), EKS (Elastic Kubernetes Service), and Lambda, making it 
easier to deploy your containerized applications in the AWS cloud.

Comparison Summary:

In short: 
• Docker is for building and running containers. 
• Amazon ECR is a cloud service to store and manage Docker container images, particularly when 
deploying to AWS infrastructure. 
Docker: 
• What it is: Docker is a platform for developing, running, and managing containers, which are 
lightweight, portable units that package applications and their dependencies. 
• Main role: It helps you build container images (using Dockerfiles) and run those containers locally or 
on any system. 
• Where it’s used: It's used on your computer or server for development, testing, and running 
applications in isolated environments. 
Amazon ECR (Elastic Container Registry): 
• What it is: ECR is a service by AWS for storing and managing Docker container images in the cloud. 
• Main role: It acts as a repository to store your Docker images securely and scale them, especially for 
use with other AWS services like ECS or EKS. 
• Where it’s used: It's used to store Docker images in the AWS cloud, making it easier to deploy those 
images to AWS-based services. 
Key Difference: 
• Docker is for creating and running containers locally. 
• ECR is for storing and managing Docker images in the AWS cloud, for use in cloud-based 
applications.
